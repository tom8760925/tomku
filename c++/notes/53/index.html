C++語言的筆記-軟體設計的重點1

2025-02-05

軟體設計的重點1

這篇是C++的筆記第53篇

1.注意軟體的依賴性和耦合

在寫程式時很容易會讓韓式之間產生依賴性，有些時候依賴性太高會造成後續出現修改問題。

#include <iostream>

// c假裝是其他檔案的class
class c
{
public:
    c();
};
c::c()
{
    std::cout << "c" << std::endl;
}

// b假裝是其他檔案的class
class b
{
public:
    b();
};
b::b()
{
    std::cout << "b" << std::endl;
}

// b和c假裝是其他函式庫
class a
{
public:
    void getb();
    void getc();
};
// 使用b的函式
void a::getb()
{
    b _b;
}
// 使用b的函式
void a::getc()
{
    c _c;
}

int main()
{
    a _a;
    _a.getb();
    _a.getc();
    return 0;
}

上述的a導入b和c使用，而如果b有做修改，例如變成b(int)，這時a也要修改，mian也要跟著修改，造成需要修改兩次，同理c也是一樣情況。
這時可以選擇b和c分離，寫單獨使用入口讓其他程式使用，就不透過a使用減少依賴。
但不是所有情況都是需要這麼做，要依照不同情況去判斷。

2.不要重複

在寫程式時盡量避免寫重複程式，因為後續需要修改程式時，可能會造成麻煩。

#include <iostream>

// a抽象化圓週長
class a
{
public:
    virtual int r() = 0;
};

// b是輸出圓周長和面積
class b : public a
{
public:
    b(int _round) : round(_round) {};
    int r() override;
    int CircleArea();

private:
    int round;
};
int b::r()
{
    return this->round * 2 * 3.14;
}
int b::CircleArea()
{
    return this->round * this->round * 3.14;
}

// c是輸出圓周長和圓柱
class c : public a
{
public:
    c(int _round) : round(_round) {};
    int r() override;
    int Cylinder(int);

private:
    int round;
};
int c::r()
{
    return this->round * 2 * 3.14;
}
int c::Cylinder(int h)
{
    return this->round * this->round * h * 3.14;
}

int main()
{
    b _b(5);
    std::cout << _b.r() << std::endl;
    std::cout << _b.CircleArea() << std::endl;
    c _c(5);
    std::cout << _c.r() << std::endl;
    std::cout << _c.Cylinder(5) << std::endl;
    return 0;
}

上述是跟圓有關的程式，如果不是乘以3.14而是3.141，就會造成所有乘以3.14的函式都需要修改。
但如果將3.14分離，就會變成只需要修改一次。

3.可測試和使用

每個程式都需要進行測試，但有些情況會造成無法進行測試。

#include <iostream>
// 資料格式
struct b
{
    int time;
};
class a
{
public:
    // 設定time
    void settime(int n);

private:
    struct b _b;
    // 設定b
    void upb(int n);
};
void a::settime(int n)
{
    this->upb(n);
}
void a::upb(int n)
{
    this->_b.time = n;
}
int main()
{
    a _a;
    _a.settime(5);
    return 0;
}

上述的a的裡有個私人成員_b，而upb是更新_b的函式，為了要使用upb，所以新增settime函式使用upb。 但這樣這樣就增加依賴性，也讓封裝性降低。

#include <iostream>
// 資料格式
struct b
{
    int time;
};
// 單純負責b
class c
{
public:
    void upb(int n);
    struct b getb();

private:
    struct b _b;
};
void c::upb(int n)
{
    this->_b.time = n;
}
struct b c::getb()
{
    return this->_b;
}
// 使用c
class a
{

private:
    c _c;
};

int main()
{
    a _a;

    return 0;
}

上述是將upb外移，讓b有單純負責他的程式，這樣封裝性提高和降低依賴性，後續修改程式或是測試就能較為簡單。
在寫程式需要考慮是否需要為私人成員。

4.要擴展

程式是需要要有擴展性，但前面的還要遵守。

class a
{
public:
    int add(int, int);
    double add(double, double);
};
int a::add(int n1, int n2)
{
    return n1 + n2;
}
double a::add(double d1, double d2)
{
    return d1 + d2;
}

int main()
{
    a _a;
    return 0;
}

上述的a可以擴展，但每擴展都需要新增函數，可以使用模板去改良。
但在設計擴展前，要先知道怎麼擴展，不要盲目擴展，不知怎麼擴展可以順序排在最後。

5.高層次、低層次和中間

在各式各樣程式，可以將其劃分為高層次、低層次和中間。
1.高層次為穩定的、低依賴、不太會或不修改。 2.低層次為會修改、高依賴、經常改變。 3.中間則為兩者之間。 在設計程式時有個基準。

#include <iostream>

class a
{
public:
    a(int);
};

a::a(int n)
{
    std::cout << n << std::endl;
}
int main()
{
    a _a(1);
    return 0;
}

在上述程式中，高層次為iostream，因為不會去修改也穩定。低層次為main，因為會修改經常改變。而中間為a，因為在兩者之間。
